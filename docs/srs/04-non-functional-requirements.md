# 4. 비기능 요구사항

## 4.1 성능 (NFR-100)

### NFR-101: 초기 로딩 시간

- 애플리케이션 초기 로딩은 **3초 이내**에 완료되어야 한다 (로컬 환경 기준).
- Monaco Editor는 CDN 로딩 방식을 사용하여 번들 크기를 최소화한다.
- 코드 스플리팅을 적용하여 초기 번들에 필수 컴포넌트만 포함한다.

### NFR-102: 터미널 입력 지연

- 사용자 키 입력부터 화면 반영까지 **16ms 이내**를 목표로 한다 (60 FPS 동기화).
- 터미널 출력 배치 처리: **16ms 단위**로 묶어 렌더링한다.

### NFR-103: 파일 트리 성능

- **10,000개 파일**이 있는 프로젝트에서도 부드러운 스크롤(60 FPS)을 유지해야 한다.
- 가상화 렌더링으로 화면에 보이는 노드만 DOM에 생성한다.

### NFR-104: 프리뷰 갱신 속도

- 에디터 변경 후 프리뷰 갱신까지 **500ms 이내**에 완료되어야 한다.
  - Markdown → HTML 변환: 300ms 이내
  - HTML iframe 업데이트: 100ms 이내
  - PDF 페이지 렌더링: 500ms 이내 (단일 페이지)

### NFR-105: WebSocket 재연결

- 연결 끊김 후 자동 재연결을 시도해야 한다.
- 지수 백오프: 1초 시작, **30초 상한**.
- 재연결 상태를 사용자에게 시각적으로 표시한다.

### NFR-106: 에디터 탭 전환

- 탭 전환 시 **50ms 이내**에 새 파일 콘텐츠가 표시되어야 한다.
- Monaco 모델은 메모리에 캐시하여 재생성을 방지한다.

### NFR-107: 메모리 관리

- 장시간 사용 시에도 메모리 누수가 없어야 한다.
- 터미널 스크롤백 버퍼: 최대 **10,000줄** (초과 시 오래된 데이터 폐기).
- 닫힌 탭의 Monaco 모델은 일정 시간 후 해제한다.

## 4.2 보안 (NFR-200)

### NFR-201: 네트워크 바인딩

- 서버는 기본적으로 `127.0.0.1` (localhost)에만 바인딩해야 한다.
- 원격 접근은 SSH 터널 또는 Cloudflare Tunnel을 통해서만 허용한다.
- 외부 노출 시 토큰 기반 인증을 강제한다.

### NFR-202: 경로 순회 방지

- 모든 파일 시스템 API에서 `path.resolve()` 기반 바운드 체크를 수행해야 한다.
- 프로젝트 루트 디렉토리 외부 경로 요청은 `403 Forbidden`으로 응답한다.

### NFR-203: iframe 샌드박스

- HTML 프리뷰 iframe에 `sandbox="allow-scripts"` 속성을 적용해야 한다.
- `allow-same-origin`은 **절대** 사용하지 않는다.
- `referrerpolicy="no-referrer"`를 적용한다.

### NFR-204: dotfile 접근 차단

- `.env`, `.git`, `.claude`, `.ssh` 등 민감한 dotfile/dot 디렉토리 접근을 API 레벨에서 차단해야 한다.

### NFR-205: 심볼릭 링크 검증

- 파일 접근 시 `fs.lstat()`로 심볼릭 링크 여부를 확인해야 한다.
- 심볼릭 링크가 프로젝트 루트 외부를 가리키면 접근을 거부한다.

### NFR-206: 파일 크기 제한

- 읽기/쓰기 API에 파일 크기 제한을 적용해야 한다.
  - 텍스트 파일: 최대 **10MB**
  - 바이너리 파일: 최대 **50MB**
- 제한 초과 시 `413 Payload Too Large`로 응답한다.

### NFR-207: 요청 속도 제한

- 파일 시스템 API에 속도 제한(rate limiting)을 적용해야 한다.
- IP당 분당 최대 **300 요청** (조정 가능).

### NFR-208: XSS 방지

- Markdown 렌더링 시 sanitize 옵션을 필수 적용한다.
- 사용자 입력을 HTML에 삽입할 때 이스케이프 처리한다.
- `dangerouslySetInnerHTML` 사용을 금지한다.

### NFR-209: CLI 명령 인젝션 방지

- 사용자 입력을 Claude CLI에 전달할 때 JSON 캡슐화를 사용한다.
- 셸 메타문자 (`|`, `;`, `&&`, `` ` `` 등)를 직접 전달하지 않는다.

## 4.3 사용성 (NFR-300)

### NFR-301: VS Code 유사 키바인딩

- 기본 키바인딩은 VS Code와 동일하게 설정해야 한다.
- `Cmd+P`, `Cmd+B`, `Cmd+J`, `Cmd+S`, `Cmd+K` 등 주요 단축키 지원.

### NFR-302: 테마 지원

- 다섯 가지 테마를 제공해야 한다: **다크 (기본)**, **라이트**, **고대비**, **Retro — Green Phosphor** (v0.3), **시스템** (v0.5).
- **시스템** 테마는 `window.matchMedia('(prefers-color-scheme: dark)')` 를 감지하여 OS의 다크/라이트 설정을 자동으로 추종한다. OS 설정이 변경되면 실시간으로 테마가 전환된다.
- 각 테마 클래스에 `color-scheme` CSS 속성을 설정하여 네이티브 스크롤바·폼 컨트롤 등 브라우저 기본 UI 요소가 앱 테마와 일치하도록 한다(OS 모드와 독립).
- FOUC(Flash of Unstyled Content) 방지를 위해 `layout.tsx`에 인라인 스크립트를 삽입하여 React 하이드레이션 전에 `localStorage`에서 테마를 읽어 적용한다.
- 테마 전환 시 모든 패널(에디터, 터미널, 파일 트리, 프리뷰)이 일관되게 변경된다.
- Retro 테마는 옵션으로 CRT 스캔라인 오버레이를 활성화할 수 있으며 (`retroScanlines` 플래그), 기본 폰트는 `VT323` → `IBM Plex Mono` → `monospace` 순서로 폴백한다.
- 테마 선택은 커맨드 팔레트 또는 설정 UI에서 가능해야 한다.

### NFR-303: WAI-ARIA 접근성

- 주요 UI 요소에 WAI-ARIA 속성을 적용해야 한다.
- 시맨틱 마크업을 사용한다.
- 스크린 리더 호환성을 보장한다.

### NFR-304: 키보드 네비게이션

- 모든 주요 기능을 키보드만으로 조작할 수 있어야 한다.
- 포커스 순서: 파일 탐색기 → 에디터 → 터미널 → 프리뷰
- 포커스 이동 단축키를 제공한다.

### NFR-305: 고대비 테마

- 시각 접근성을 위한 고대비 테마 옵션을 제공해야 한다.

### NFR-306: 폰트 크기 조절

- 에디터 및 터미널의 폰트 크기를 사용자가 조절할 수 있어야 한다.
- `Cmd+`/`Cmd-`로 빠른 조절을 지원한다.

### NFR-307: API 에러 사용자 알림

- API 호출 실패(네트워크 오류, 서버 오류 등)는 토스트 알림으로 사용자에게 즉시 표시해야 한다.
- 토스트는 5초 후 자동 해제되며, 수동으로도 닫을 수 있다.
- 최대 5개의 토스트가 동시에 표시되며, 오래된 것은 자동으로 제거된다.
- 구현: `src/stores/use-toast-store.ts`, `src/components/layout/toast-container.tsx`, `src/lib/api-client.ts` (`showErrorToast`)

## 4.4 호환성 (NFR-400)

### NFR-401: 브라우저 지원

- **Chrome** 최신 2개 버전을 공식 지원해야 한다.
- Firefox, Safari, Edge는 베스트 에포트로 지원한다.

### NFR-402: 운영체제 지원

- macOS 13+, Windows 10+, Ubuntu 20.04 LTS+에서 동작해야 한다.
- `node-pty` 네이티브 모듈이 각 OS에서 빌드 가능해야 한다.

### NFR-403: 최소 해상도

- **1280 × 720** 이상 해상도를 지원해야 한다.
- 뷰포트 너비 1280px 미만에서는 **단일 패널 탭 모드**로 폴백한다. 하단에 탭 바(Files, Editor, Terminal, Claude, Preview)를 표시하여 패널 간 전환을 제공하며, 한 번에 하나의 패널만 전체 화면으로 표시한다.

### NFR-404: Node.js 호환성

- Node.js **20.x–24.x** (LTS **22.x** 권장) 을 지원해야 한다.
- `@parcel/watcher` v2 네이티브 바이너리 / node-pty 프리빌트 / ESM dynamic import가 모두 동작하는 런타임이어야 한다.

## 4.5 유지보수성 (NFR-500)

### NFR-501: TypeScript Strict 모드

- `strict: true` 설정으로 타입 안전성을 보장해야 한다.
- `any` 타입 사용을 금지한다.

### NFR-502: 테스트 커버리지

- 컴포넌트 및 유틸리티 함수의 테스트 커버리지 **70% 이상**을 유지해야 한다.
- 핵심 모듈(파일 시스템 API, WebSocket 핸들러, 권한 관리)은 **90% 이상**.

### NFR-503: 코드 품질

- ESLint + Prettier를 적용하여 일관된 코드 스타일을 유지해야 한다.
- CI 파이프라인에서 린팅 및 포맷 검사를 자동 실행한다.

### NFR-504: 모듈화

- 각 패널 컴포넌트는 독립적으로 테스트 가능해야 한다.
- Zustand 스토어는 슬라이스 패턴으로 분리한다.
- 서버 측 핸들러(터미널, Claude, 파일)는 독립 모듈로 구성한다.

### NFR-505: 문서화

- 공개 API(REST, WebSocket)는 문서화해야 한다.
- 아키텍처 결정(ADR)은 기록을 유지한다.
