## Architecture Decisions (ADR) Summary

| ID | Decision | Rationale |
|----|----------|-----------|
| ADR-001 | `ws` library (instead of socket.io) | Low overhead (~5 KB vs 10.4 KB); standards-compliant WebSocket |
| ADR-002 | Use the Agent SDK (instead of child_process) | Avoids CLI hang issues; type safety; session management |
| ADR-003 | Monaco Editor (instead of CodeMirror) | VS Code-level features; 100+ languages; developer familiarity |
| ADR-004 | `react-resizable-panels` | 5.2k stars; localStorage support; collapse/expand |
| ADR-005 | Zustand (instead of Redux) | Lightweight; accessible outside React; minimal boilerplate |
| ADR-006 | reveal.js (instead of Marp) | 70k stars; rich API; live editing via `Reveal.sync()` |
| ADR-007 | Custom Node.js server (instead of Vercel) | WebSocket required; long-lived process support |
| ADR-008 | iframe srcdoc (instead of eval) | Sandboxing security; postMessage communication |
| ADR-009 | Use `git` CLI for Git status | Avoids isomorphic-git bundle cost; performance and simplicity — `src/lib/fs/git-status.ts` |
| ADR-010 | Separate `/api/files/raw` binary endpoint | Dedicated to image/PDF streaming; security boundary separate from text READ |
| ADR-011 | Agent SDK `canUseTool` callback for permission handling | Uses the official SDK option instead of intercepting the tool_use stream — stable event ordering and aborted-signal handling. Implementation: `server-handlers/claude-handler.mjs` |
| ADR-012 | `didWebSocketSetup = true` flag to block NextCustomServer's own upgrade listener | Prevents conflicts between Next 14's internal WebSocket setup and ClaudeGUI's `/ws/*` routing. Revalidate on a major Next upgrade. Implementation: `server.js` |
| ADR-013 | Singleton `/ws/files` and `/ws/claude` clients | Ensures a single WS connection is maintained even when multiple hooks on the page call in. `getFilesClient()`, `getClaudeClient()` — `src/lib/websocket/*-client.ts`. Connection status is unified in `useConnectionStore`. |
| ADR-014 | Line-based LCS diff with per-hunk partial acceptance | A pure JS LCS implementation instead of the Monaco DiffEditor API. Consistent server/client results, easy to test. Implementation: `src/lib/diff/line-diff.ts`, `toggleHunk`/`applyAcceptedHunks` in `src/stores/use-editor-store.ts` |
| ADR-015 | UI to edit `.claude/settings.json`'s `permissions.allow`/`deny` | `canUseTool` handles only user-prompt approval; persistent auto-approvals are configured via Claude Code's standard rules in `settings.json`. Parse/save: `src/lib/claude/settings-manager.ts` |
| ADR-016 | `ProjectContext` singleton for runtime project hot-swap | `PROJECT_ROOT`, previously frozen at boot, becomes runtime-switchable. `src/lib/project/project-context.mjs` is the single source of truth; the files/terminal/claude handlers and `resolveSafe` all call `getActiveRoot()`. On change: the file watcher (since ADR-024, `@parcel/watcher`) is re-subscribed and a `project-changed` event is broadcast over WS. State persists to `~/.claudegui/state.json`. Now wrapped by `BrowserSessionRegistry` (ADR-027) for per-tab project isolation; the singleton serves as the fallback when no `browserId` is present. |
| ADR-017 | HTML stream extractor (state machine) | Watches Claude assistant text for ` ```html ` fences and `Write`/`Edit` `tool_use` events, pushing partial HTML into a live preview store. A renderability heuristic (`<!doctype` or balanced tags) chooses iframe srcdoc vs. source-view fallback. Implementation: `src/lib/claude/html-stream-extractor.ts`, `src/stores/use-live-preview-store.ts`. |
| ADR-018 | Tauri v2 + Node sidecar native installer | Chosen over Electron: native webview + bundled Node.js sidecar running `server.js`. `.dmg`/`.msi` are 40–60 MB, and native ABIs like node-pty are rebuilt in CI against the bundled Node version. On first launch the app installs `@anthropic-ai/claude-code` into an app-local `node-prefix` and prepends its `bin` to the PTY `PATH`. Implementation: `installer/tauri/`, `scripts/installer-runtime/ensure-claude-cli.mjs`. |
| ADR-019 | Terminal session durability — login+interactive shells, no auto-reconnect, restart-in-place | (a) Spawn the PTY with `['-l','-i']` so user dotfiles are sourced (FR-410). (b) ~The terminal WebSocket does not auto-reconnect~ → **superseded by ADR-020**. (c) ~No server-side session registry is introduced~ → **superseded by ADR-020**. (d) No automatic `cd` is injected into existing tabs on project switch — risks clobbering in-progress shell state. A banner surfaces the mismatch and offers "Open new tab here" instead. (e) cwd tracking is implemented by the server writing an OSC 7 emitter snippet directly to PTY stdin immediately after spawn, followed by `clear` and a ring buffer reset so the injection is invisible to the user. Implementation: `server-handlers/terminal/shell-resolver.mjs`, `server-handlers/terminal-handler.mjs`, `src/lib/terminal/terminal-socket.ts`, `src/lib/terminal/terminal-manager.ts` (`restartSession`), `src/components/panels/terminal/terminal-panel.tsx`. |
| ADR-020 | Server-side terminal session registry with reconnect replay (supersedes ADR-019 (b)/(c)) | Introduced in v0.5 at user request. PTYs are registered in an in-memory `TerminalSessionRegistry`; on WebSocket disconnect the handler detaches instead of killing the process. A 30-minute grace timer protects the session, and reconnecting with `?sessionId=<uuid>` replays a 256 KB ring buffer via a `{type:"session",replay:true}` control frame followed by a single binary frame so the client's xterm buffer is restored. The tab close button sends `{type:"close"}` to destroy immediately. Restart-in-place reuses the same path (drop socket → reconnect with same id → replay). All sessions are lost if the server process restarts (persistence is future work). Rationale: HMR cycles and browser reloads were repeatedly dropping shell state in real workflows, which re-opened the simplicity-vs-durability tradeoff from ADR-019. Implementation: `server-handlers/terminal/session-registry.mjs`, `server-handlers/terminal-handler.mjs`, `src/lib/terminal/terminal-manager.ts` (`buildSocketUrl`, the `session` case in `applyServerControl`, the `{type:"close"}` frame in `closeSession`), `src/lib/terminal/terminal-framing.ts` (`TerminalCloseControl`, `TerminalSessionServerControl`). |
| ADR-021 | Explorer-driven dynamic re-rooting + removal of silent cwd fallback (extends ADR-016) | Introduced in v0.6 at user request. (a) The `$HOME` ban in `validateProjectRoot` is removed; only the filesystem root (`/` or a Windows drive root) is still rejected — users must be able to open their home directory or any ancestor as a project. The dotfile deny list in `resolveSafe` still blocks sensitive files. (b) `resolveInitialRoot` no longer falls back to `process.cwd()` and instead returns `null`, so the server's launch directory never silently becomes the project root (which previously caused Claude to write artifacts to unexpected locations). (c) `getActiveRoot()` is now `string \| null`. When no root is active, `claude-handler` rejects `runQuery` with a `4412` error, `resolveSafe`/`getProjectRoot()` throws `SandboxError(4412)`, and the `@parcel/watcher` subscription stays idle; the terminal handler keeps its `os.homedir()` fallback. (d) `AppShell` force-opens the `ProjectPickerModal` once `useProjectStore.refresh()` resolves with a null root. (e) The file explorer UI surfaces re-rooting through an Up button, a clickable breadcrumb, and an "Open as project root" directory context menu item (`FR-209`). Implementation: `src/lib/project/project-context.mjs`, `server-handlers/claude-handler.mjs`, `src/lib/fs/resolve-safe.ts`, `server-handlers/files-handler.mjs`, `src/stores/use-project-store.ts`, `src/components/layout/app-shell.tsx`, `src/components/panels/file-explorer/file-explorer-panel.tsx`, `src/components/panels/file-explorer/file-tree.tsx`. |
| ADR-022 | Script-installed desktop launcher (`.command` / `.desktop` / `.lnk`) — complements the Tauri native installer (ADR-018) | Gives one-line curl/iwr installer users a "double-click an icon → server + browser" GUI entry point. For simplicity, macOS uses a `.command` file instead of an `.app` bundle, so we avoid Gatekeeper signing entirely (at the cost of a custom shortcut icon — the default Terminal icon shows up). Linux drops a `.desktop` entry whose `Icon=` field points at the absolute SVG mascot. Windows creates a PowerShell-console-backed `.lnk` via `WScript.Shell.CreateShortcut` and sets `IconLocation` to `claudegui.ico`. The launcher script polls `localhost:$PORT` from a background job and launches the OS default browser as soon as the server is ready, while running `node server.js` in the foreground so closing the console window sends SIGHUP and stops the server. A single SVG (`public/branding/claudegui.svg`) is the source of truth for both the desktop icon and the Next.js favicon (`src/app/icon.svg`); `scripts/build-icons.mjs` rasterizes every PNG/ICO via macOS-native tools (`qlmanage` / `sips`) and a small in-script PNG-in-ICO packer. Where ADR-018 (Tauri) covers the native `.dmg`/`.msi` user path, ADR-022 covers the source-install user path. Implementation: `scripts/install/install.sh`, `scripts/install/install.ps1`, `scripts/build-icons.mjs`, `public/branding/`, `src/app/icon.svg`, `src/app/apple-icon.png`. |
| ADR-023 | Hoisted single-instance file context menu + in-app clipboard (FR-206/FR-211) | Introduced in v0.7 at user request. (a) **Problem**: with a per-node `<Radix ContextMenu>` on a react-arborist virtualized tree, opening a context menu and then moving the mouse causes the tree to re-render and replace the trigger DOM, dismissing the menu instantly. Per-node menus also make multi-selection ambiguous about which node owns the menu. (b) **Decision**: mount exactly one context menu at the panel root (`file-context-menu.tsx`); each node's `onContextMenu` calls `useFileContextMenuStore.openAtNode({ clientX, clientY }, target, selectionPaths)` to publish coordinates, target, and selection. The menu is a Radix `DropdownMenu` with controlled `open`, anchored to a 1×1 invisible `position: fixed` button moved to the click coordinates. (c) **Result**: virtualization re-rendering and hover updates can no longer destroy the menu trigger, so the menu stays open while the user is mousing toward an item. The menu only closes on `Esc`, an outside click, or a context-menu re-open on a different node (anchor swap). (d) **Extension**: an in-app clipboard (`useFileClipboardStore` with `mode: 'copy' \| 'cut'`) and a `POST /api/files/copy` route (`fs.cp` plus ` (n)` collision avoidance) sit on the same hoisted pattern, so Cut/Copy/Paste/Duplicate share a single source of truth across the context menu, keyboard shortcuts (`use-file-keyboard.ts`), and Alt-drag move/copy. Cut nodes render with `italic + opacity-50`. (e) **Rejected alternatives**: a fully custom floating menu (would have to re-implement Radix's a11y/focus trap/Escape handling); adding `forceMount` to the per-node ContextMenu (does not stop virtualization re-renders, only delays the symptom). Implementation: `src/components/panels/file-explorer/file-context-menu.tsx`, `src/components/panels/file-explorer/use-file-actions.ts`, `src/components/panels/file-explorer/use-file-keyboard.ts`, `src/components/panels/file-explorer/delete-confirm-dialog.tsx`, `src/stores/use-file-context-menu-store.ts`, `src/stores/use-file-clipboard-store.ts`, `src/components/ui/dropdown-menu.tsx`, `src/app/api/files/copy/route.ts`, `src/lib/fs/file-operations.ts` (`copyEntry`, recursive `deleteEntry`). |
| ADR-024 | Replace chokidar v5 with `@parcel/watcher` v2 | (a) **Problem**: chokidar dropped its native fsevents dependency in v4 (and v5 kept the change), so on macOS it falls back to `fs.watch`, which consumes **one file descriptor per watched subdirectory**. Combined with node-pty sessions, WS connections, and the Next dev compile worker, ClaudeGUI runs into macOS's default **256 FD per-process** soft limit (inherited from launchd), and `files-handler` emits an endless `EMFILE: too many open files, watch` loop that effectively kills the watcher. No amount of ignore-glob tuning fixes this — the crash reproduced even with `node_modules`/`.next`/`.git` already excluded. (b) **Decision**: switch the file watcher to `@parcel/watcher` v2, which sits on top of native FSEvents (macOS) / inotify (Linux) / ReadDirectoryChangesW (Windows) and uses **one OS handle per watched root**. The `ignore` option accepts micromatch globs (`**/node_modules`, `**/node_modules/**`, …) and prunes the heavy subtrees at the native layer so they're never even scanned. The event model collapses to `create`/`update`/`delete`; since downstream consumers (editor reload, tree refresh, git-status refresh) don't need per-type fidelity, these are normalized to `add`/`change`/`unlink` with additional `ready`/`error` signals. The segment-anchored JS filter `isIgnoredByWatcher` is preserved as a broadcast-side dotfile/`.DS_Store` guard. (c) **Rejected alternatives**: (i) raising the soft FD limit via a `ulimit` re-exec — treats the symptom but chokidar would still burn FDs linearly with tree size; (ii) calling Node's native `fs.watch(path, { recursive: true })` directly — also FSEvents-backed on macOS but has a coarser event model and we'd have to reimplement ignore matching ourselves; `@parcel/watcher` already does that on top of the exact same OS APIs; (iii) downgrading to chokidar 3 — out of support, does not receive security/bug fixes. (d) **Implementation**: `server-handlers/files-handler.mjs` (`loadWatcher` → `subscribe`, `rebuildWatcher` → `unsubscribe`, event mapping, exported `WATCHER_IGNORE_GLOBS`); `src/lib/fs/watcher.ts` (TypeScript wrapper with the same `createWatcher` API); `next.config.js` (`@parcel/watcher` registered in `serverComponentsExternalPackages` and the webpack externals; chokidar removed); `package.json` (drop `chokidar`, add `@parcel/watcher`); `src/types/websocket.ts` (`FileChangeMessage.event` narrowed to `add`/`change`/`unlink`/`ready`/`error` with an optional `message` field); `tests/unit/files-watcher-ignore.test.ts` (added `WATCHER_IGNORE_GLOBS` coverage assertions). |

| ADR-026 | Streaming enhancements — real-time editor diff + chat UX + preview improvements (FR-312, FR-313) | Wires Claude's `Write`/`Edit`/`MultiEdit` tool_use directly to the editor via `applyClaudeEdit`/`updateStreamingEdit` to display real-time diff views. `input_json_delta` streaming accumulates tool input and updates partial diffs every 500ms, transitioning to the final diff on `content_block_stop`. `syncExternalChange` uses a `tab.diff` guard to prevent race conditions. Chat panel additions include a thinking indicator (empty message + blinking cursor), tool_use file path display (click to open in editor), `useDeferredValue`-based ReactMarkdown performance optimization, and a `StreamingActivityBar`. Preview debounce is reduced to HTML 80ms / Markdown 100ms with a shimmer progress bar. Auto panel expansion ensures editor/preview panels open automatically when related content arrives. Implementation: `src/stores/use-claude-store.ts`, `src/stores/use-editor-store.ts`, `src/components/panels/editor/diff-accept-bar.tsx`, `src/components/panels/claude/chat-message-item.tsx`, `src/components/panels/claude/claude-chat-panel.tsx`, `src/components/panels/preview/live-stream-preview.tsx`, `tailwind.config.ts`. |
| ADR-025 | MCP server integration — external servers via settings.json (FR-1400) | External stdio/SSE/HTTP servers are stored per-project in `.claude/settings.json`'s `mcpServers` field. `claude-handler.mjs`'s `runQuery()` loads enabled servers and passes them to `queryOptions.mcpServers`; the SDK manages process lifecycle. MCP tool calls pass through the existing `canUseTool` callback (ADR-011) — no separate logic needed. **UI**: Zustand store + modal + header Blocks button + status bar indicator. Rejected alternative: global config — different projects need different tools. | ADR-028 | Panel-level error boundary isolation (NFR-506) | (a) **Problem**: in the 4-panel layout, a render error in any panel propagates up the React tree and blanks the entire app. There is no production visibility — the user sees a blank screen. (b) **Decision**: `src/components/layout/error-boundary.tsx` ships a generic `<ErrorBoundary scope>` and a panel-scoped `<PanelErrorBoundary panelType>`. `leaf-panel.tsx::renderPanel()` wraps each mounted panel with `PanelErrorBoundary`, and `app-shell.tsx` wraps the root with `ErrorBoundary`. Fallbacks expose a retry button plus the error message. A pluggable `registerErrorSink(fn)` API lets us attach Sentry/telemetry later; the default sink is `console.error`. (c) **Rejected**: react-error-boundary dependency — current requirements (scope tagging + pluggable sink) fit in a single source file. (d) Implementation: `src/components/layout/error-boundary.tsx`, `src/components/layout/leaf-panel.tsx`, `src/components/layout/app-shell.tsx`, tests `tests/unit/error-boundary.test.tsx`. |
| ADR-029 | Request aborter registry to eliminate `closeTab` races (NFR-507) | (a) **Problem**: `use-claude-store.closeTab` called `import('@/lib/websocket/claude-client').then(...).abort(requestId)` inside the Zustand reducer, which deferred the abort to a microtask and left its ordering relative to `requestToTabMap.delete` unclear. Side effects inside a reducer are also re-entrancy hazards. (b) **Decision**: add `src/lib/claude/request-aborter.ts` with a minimal `registerAborter(fn)` / `abortRequest(id)` registry. `getClaudeClient()` registers its `abort` method during construction. `closeTab` now calls `abortRequest(reqId)` synchronously *outside* the reducer, clears `requestToTabMap`, then invokes `set(...)`. This breaks the store ↔ client import cycle and guarantees abort runs before tab state is removed. (c) Implementation: `src/lib/claude/request-aborter.ts` (new), `src/lib/websocket/claude-client.ts`, `src/stores/use-claude-store.ts`. Tests: `tests/unit/claude-store.test.ts` with two closeTab-during-streaming cases. |
| ADR-031 | `use-claude-store` module decomposition (reinforces NFR-504) | (a) **Problem**: a single 1,333-line file mixed type declarations, pure helpers, module-level `Map`s, stream extractor lifecycle code, and Zustand actions. This made targeted testing and incremental refactoring hard. (b) **Decision**: a behavior-preserving split that keeps the public API surface and actions unchanged. (c) **Result**: `use-claude-store.ts` shrinks from 1,333 → 1,060 lines (−273, −20%); all existing tests (13) still pass. Slicing the 444-line `handleServerMessage` dispatcher is deferred to a follow-up. (d) Implementation: `src/stores/claude/types.ts`, `src/stores/claude/helpers.ts`, `src/stores/claude/extractors.ts`, `src/stores/use-claude-store.ts`. |
| ADR-032 | Phase 2 performance pass — remove streaming O(n²), isolate re-renders, GC buckets | (a) **Problems**: (i) the `content_block_delta` path in `handleServerMessage` ran `streamingChunks.push` followed by `chunks.join('')` on every token, producing O(n²) work over the full response. (ii) `StreamingActivityBar` received `messages` as a prop and scanned the full array on every token commit. (iii) A 1-second `setInterval` in `FileExplorerPanel` re-rendered the whole panel once per second. (iv) `rate-limit.ts` never evicted expired buckets, so a long-lived server accumulated memory across unique clients. (b) **Decisions**: (i) switch to incremental string concat (`last.content + delta`); `streamingChunks` is still maintained as a buffer for consumers that snapshot the raw deltas, but the re-join is eliminated — total build work is now amortized O(n). (ii) `StreamingActivityBar` now takes only `tabId` as a prop and subscribes directly via `useShallow(useClaudeStore, …)` — it re-renders only when `toolName` or `filePath` actually change, not on every reshaped messages array. (iii) extracted `<RelativeTime timestamp>` (`src/components/ui/relative-time.tsx`) that owns its `now` state and pauses its timer via the Page Visibility API when the document is hidden. The parent panel no longer re-renders per tick. (iv) `rate-limit.ts` adds a 5-minute throttled sweep that removes expired buckets on the next inbound call. (c) **Rejected / deferred**: the live-preview iframe `postMessage` partial-update plan (Phase 2.2) is deferred — it needs browser-integration tests before shipping, and the current 150ms debounce bounds the practical impact. (d) Implementation: `src/stores/use-claude-store.ts` (incremental concat), `src/components/panels/claude/claude-chat-view.tsx` (StreamingActivityBar), `src/components/ui/relative-time.tsx` (new), `src/components/panels/file-explorer/file-explorer-panel.tsx`, `src/lib/fs/rate-limit.ts`. Tests: `tests/unit/rate-limit.test.ts` (8), `tests/unit/relative-time.test.tsx` (7), `tests/unit/claude-store.test.ts` (+1 streaming incremental build). |

| ADR-027 | Multi-browser independent project support via `BrowserSessionRegistry` (FR-1500) | (a) **Context**: the existing `ProjectContext` singleton (ADR-016) shares one project root across all browser tabs connected to the same server.
