| ADR-024 | 파일 감시기를 chokidar v5 → `@parcel/watcher` v2로 교체 | (a) **문제**: chokidar는 v4부터 네이티브 fsevents 의존성을 제거했고 v5도 그대로 유지했다. 그 결과 macOS에서 `fs.watch` 폴백으로 동작하며 **서브디렉토리마다 파일 디스크립터 1개**를 소비한다. ClaudeGUI 프로젝트의 watchable 디렉토리 수(77+)에 더해 node-pty 세션, WS 연결, Next dev 컴파일 worker 등이 FD를 공유하고, macOS가 launchd로부터 상속하는 기본 **256 FD/프로세스** 소프트 한도에 도달하면 `files-handler`가 `EMFILE: too many open files, watch`를 끝없이 방출하고 워처가 사실상 죽는다. 무거운 디렉토리에 대한 ignore 글롭으로는 근본적으로 해결되지 않는다(이미 ignore 중인 프로젝트에서도 재현). (b) **결정**: `@parcel/watcher` v2로 전환한다. 네이티브 FSEvents(macOS) / inotify(Linux) / ReadDirectoryChangesW(Windows) API 위에서 동작하며 **루트당 OS 핸들 1개**만 사용한다. `mod.subscribe(root, cb, { ignore })`의 `ignore`는 글롭(`**/node_modules`, `**/node_modules/**` 등)을 받아 해당 서브트리를 네이티브 계층에서 아예 스캔하지 않는다. 이벤트 모델은 `create`/`update`/`delete` 단순 3종이며, 다운스트림(에디터 리로드, 트리 리프레시, git status 리프레시)이 세부 타입을 요구하지 않으므로 `add`/`change`/`unlink` + `ready`/`error`로 정규화한다. 기존의 세그먼트 앵커링된 JS 필터(`isIgnoredByWatcher`)는 dotfile/ .DS_Store 등을 broadcast 단계에서 숨기기 위해 그대로 유지한다. (c) **거부된 대안**: (i) 소프트 FD 한도 상향 재실행(ulimit re-exec) — 증상 완화일 뿐 chokidar는 여전히 디렉토리마다 FD를 소모하므로 규모에 따라 다시 한도에 부딪힌다.
